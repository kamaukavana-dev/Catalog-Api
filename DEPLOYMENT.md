# Deployment Guide

This document describes the process for deploying the Catalog API.

## 1. Build Artifact

The primary deployment artifact is a self-contained executable JAR file generated by the Spring Boot Maven plugin. To build the production-ready artifact, run:

```sh
mvn clean install
```

The JAR file will be located at `target/catalog-api-0.0.1-SNAPSHOT.jar`.

## 2. Containerization

The application is designed to be deployed as a Docker container. A `Dockerfile` is provided in the root of the repository to build the container image.

### Building the Docker Image

To build the image, run the following command from the project root:

```sh
docker build -t catalog-api:latest .
```

## 3. Runtime Environment

The Catalog API requires the following backing services to be available in its runtime environment:

- **PostgreSQL Database**: A running PostgreSQL instance.
- **Redis**: A running Redis instance for caching.
- **(Optional) OpenTelemetry Collector**: An endpoint for the OTLP exporter to send trace data.

## 4. Configuration

The application is configured via Spring profiles. The `prod` profile should be activated for production deployments.

### Activating the Production Profile

The `prod` profile can be activated by setting the `SPRING_PROFILES_ACTIVE` environment variable:

```
SPRING_PROFILES_ACTIVE=prod
```

### Environment Variables

The following environment variables must be configured for a production deployment to override the defaults in `application-prod.yml`:

- `SPRING_DATASOURCE_URL`: The JDBC URL of the PostgreSQL database.
- `SPRING_DATASOURCE_USERNAME`: The database username.
- `SPRING_DATASOURCE_PASSWORD`: The database password.
- `SPRING_DATA_REDIS_HOST`: The hostname of the Redis server.
- `SPRING_DATA_REDIS_PORT`: The port of the Redis server.

Security and storage configuration:
- `CATALOG_REQUIRE_API_KEY`: Set to `true` to enforce API keys for mutation endpoints.
- `CATALOG_API_KEYS`: Comma-separated list of allowed API keys.
- `S3_BUCKET_NAME`, `S3_ENDPOINT`, `S3_BASE_URL`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`.

Other configurations related to logging, tracing, and AWS can also be overridden via environment variables as needed.

## 5. Deployment Strategy

**NOT VERIFIED FROM CODEBASE.**

While the application is architected for it, a specific deployment strategy (e.g., Kubernetes, ECS, serverless) is not defined in this repository.

A typical deployment to a container orchestration platform like **Kubernetes** would involve:
1.  Pushing the Docker image to a container registry (e.g., Docker Hub, ECR, GCR).
2.  Creating a Kubernetes Deployment manifest to manage the application pods.
3.  Creating a Kubernetes Service manifest to expose the application pods behind a stable endpoint.
4.  Using Kubernetes Secrets to manage database credentials and other sensitive information.
5.  Configuring a Horizontal Pod Autoscaler (HPA) to automatically scale the number of pods based on CPU or memory usage.
